## Approach
A Kafka to Flink migration is challenging primarily because streaming pipelines have very different semantics from batch. There's no natural "end of the run" to validate against, so correctness checks have to be redesigned around continuous event time, late-arriving data, and exactly-once or at-least-once delivery guarantees. Backpressure, state size, and checkpoint tuning also become ongoing operational concerns rather than a one-time job config.

## Technique / Data Structure / Algorithm Used
- Stream processing semantics (event time vs processing time, watermarks, windowing)
- Exactly-once semantics via checkpointing and idempotent sinks (Flink two-phase commit)
- Dual-write / shadow traffic reconciliation (CDC + parallel topics with diffing)
- Backpressure handling and state backend tuning (RocksDB, incremental checkpoints)

## Code
```python
# Example: Flink job skeleton consuming Kafka, windowing, writing to sink with EOS
from pyflink.datastream import StreamExecutionEnvironment, CheckpointingMode
from pyflink.datastream.connectors.kafka import KafkaSource, KafkaSink, DeliveryGuarantee
from pyflink.datastream.window import TumblingEventTimeWindows
from pyflink.common import Duration, WatermarkStrategy

env = StreamExecutionEnvironment.get_execution_environment()
env.enable_checkpointing(60_000, CheckpointingMode.EXACTLY_ONCE)

# Watermark strategy for ~30s out-of-orderness
ws = (WatermarkStrategy
      .for_bounded_out_of_orderness(Duration.of_seconds(30))
      .with_idle_timeout(Duration.of_seconds(60)))

source = (KafkaSource.builder()
          .set_bootstrap_servers("broker:9092")
          .set_group_id("migration-v2")
          .set_topics("events.v1")
          .set_value_only_deserializer(SimpleStringSchema())
          .build())

sink = (KafkaSink.builder()
        .set_bootstrap_servers("broker:9092")
        .set_topic("events.v2")
        .set_delivery_guarantee(DeliveryGuarantee.EXACTLY_ONCE)
        .build())

(env.from_source(source, ws, "kafka-src")
   .key_by(lambda e: e["user_id"])
   .window(TumblingEventTimeWindows.of(Duration.of_minutes(5)))
   .reduce(lambda a, b: merge(a, b))
   .sink_to(sink))

env.execute("hadoop-to-flink-migration")
```

## Dry Run
Input: events from `events.v1` keyed by `user_id` with timestamps spanning 14:00-14:05 and some stragglers up to 30s late. The source assigns watermarks based on event time. Records are partitioned by `user_id`, bucketed into 5-minute tumbling windows, merged via the reducer, and committed to `events.v2` with two-phase commit. During the 3-week dual-run, a separate reconciliation job diffs aggregate counts from both topics. When diff stays below threshold for N consecutive checkpoints, cutover is approved.

## Complexity
- Time Complexity: O(n) per record ingestion; window state grows O(k) where k = distinct keys per window.
- Space Complexity: O(state) dominated by keyed window state in RocksDB; checkpoint overhead roughly proportional to incremental state delta.
