--- Raw model answer ---
Likely Cause:
A memory leak in a long-running consumer process is usually caused by **unbounded growth of internal data structures** (caches, buffers, message maps) or **event-listener / subscription handles that are never released**, so the heap keeps climbing across each iteration.

How I Would Investigate:
- Pull a heap dump during a steady-state run and again after the leak has grown, then diff them to see which object type is accumulating.
- Capture GC logs and watch the old-gen occupancy curve after each full GC. A leak shows up as a rising baseline that never recovers.
- Use a profiler (pprof, async-profiler, dotMemory, etc.) and trigger allocations on suspect paths to confirm what is retaining the references.

Fix:
Identify the offending holder and bound it. Common remediations are switching to an LRU/TTL-bounded cache, fixing a missing `removeListener` / `close()` / `dispose()` in shutdown paths, or clearing per-batch accumulators after processing. Ship behind a flag so you can watch the heap curve flatten in production.

Validation:
Before the fix, baseline old-gen usage over a 24-hour soak test. After deploying, re-run the same soak and confirm the heap stabilizes with no monotonic growth across multiple full GCs. A short load test that previously caused OOM should now run cleanly.

Prevention:
- Add a long-running soak test (12 to 24 hours) to CI for any service that holds state.
- Wire JVM/runtime metrics (heap after GC, open handles, buffer sizes) into dashboards with alerts on trend, not just threshold.
- Enforce bounded collections in code review and require explicit close/cleanup on resources held by long-lived components.
