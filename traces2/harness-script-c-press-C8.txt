--- Raw model answer ---
Here's my approach: a Flink job that ingests ledger events from Kafka, joins them with the current balance, and produces a reconciliation record with status (MATCH, MISSING_DEBIT, MISSING_CREDIT, AMOUNT_MISMATCH). I'll show the core stream processor with a keyed process function that maintains per-account state, exactly-once semantics via checkpointing, and a sink to PostgreSQL.

## Approach

- Read ledger events from Kafka as the source.
- Key by `account_id` so all events for one account land on the same parallel instance (required for stateful processing).
- Use a `KeyedProcessFunction` with a `ValueState` (RocksDB-backed) to hold the last known balance per account and the set of pending debit/credit pairs.
- For every event, look up the current state, decide reconciliation status, then emit a result record and update state.
- Use a two-phase commit sink for exactly-once delivery to PostgreSQL.
- Use event-time with watermarks so late events can be handled in a bounded way.

## Technique / Data Structure / Algorithm Used

- Stream processing with Apache Flink.
- Keyed state (`ValueState`, `ListState`) for per-account bookkeeping.
- Event-time semantics with watermarks for late event handling.
- Exactly-once via checkpointing + two-phase commit sink.
- Idempotent state updates to keep reconciliation deterministic.

// Maven deps: flink-streaming-java, flink-connector-kafka, flink-connector-jdbc, postgresql, jackson
import org.apache.flink.api.common.eventtime.WatermarkStrategy;
import org.apache.flink.api.common.serialization.SimpleStringSchema;
import org.apache.flink.api.common.state.ListState;
import org.apache.flink.api.common.state.ListStateDescriptor;
import org.apache.flink.api.common.state.ValueState;
import org.apache.flink.api.common.state.ValueStateDescriptor;
import org.apache.flink.api.common.typeinfo.Types;
import org.apache.flink.configuration.Configuration;
import org.apache.flink.connector.jdbc.JdbcExactlyOnceOptions;
import org.apache.flink.connector.jdbc.JdbcExecutionOptions;
import org.apache.flink.connector.jdbc.JdbcSink;
import org.apache.flink.connector.kafka.source.KafkaSource;
import org.apache.flink.connector.kafka.source.enumerator.initializer.OffsetsInitializer;
import org.apache.flink.streaming.api.CheckpointingMode;
import org.apache.flink.streaming.api.datastream.DataStream;
import org.apache.flink.streaming.api.environment.CheckpointConfig;
import org.apache.flink.streaming.api.environment.StreamExecutionEnvironment;
import org.apache.flink.streaming.api.functions.KeyedProcessFunction;
import org.apache.flink.util.Collector;
import org.apache.flink.api.java.tuple.Tuple2;

import com.fasterxml.jackson.annotation.JsonIgnoreProperties;
import com.fasterxml.jackson.databind.ObjectMapper;

import java.time.Duration;
import java.util.ArrayList;
import java.util.List;

public class LedgerReconciliationJob {

    // ---- Domain model ----
    @JsonIgnoreProperties(ignoreUnknown = true)
    public static class LedgerEvent {
        public String eventId;
        public String accountId;
        public String type;     // "DEBIT" or "CREDIT"
        public double amount;
        public long   eventTime; // epoch millis
        public String reference; // optional, e.g. transaction id
        public LedgerEvent() {}
    }

    public enum ReconStatus { MATCH, MISSING_DEBIT, MISSING_CREDIT, AMOUNT_MISMATCH }

    public static class ReconResult {
        public String accountId;
        public String reference;
        public ReconStatus status;
        public double  expectedBalance;
        public double  observedAmount;
        public long    processedAt;
        public ReconResult() {}
        public ReconResult(String a, String r, ReconStatus s, double e, double o, long t) {
            accountId = a; reference = r; status = s;
            expectedBalance = e; observedAmount = o; processedAt = t;
        }
    }

    // ---- Core process function ----
    public static class ReconciliationFunction
            extends KeyedProcessFunction<String, LedgerEvent, ReconResult> {

        private transient ValueState<Double> balanceState;
        private transient ListState<LedgerEvent> pendingDebits;
        private transient ListState<LedgerEvent> pendingCredits;
        private final ObjectMapper json = new ObjectMapper();
        private final long lateThresholdMs;

        public ReconciliationFunction(long lateThresholdMs) {
            this.lateThresholdMs = lateThresholdMs;
        }

        @Override
        public void open(Configuration parameters) {
            balanceState = getRuntimeContext().getState(
                    new ValueStateDescriptor<>("balance", Types.DOUBLE));
            pendingDebits = getRuntimeContext().getListState(
                    new ListStateDescriptor<>("pendingDebits", Types.POJO));
            pendingCredits = getRuntimeContext().getListState(
                    new ListStateDescriptor<>("pendingCredits", Types.POJO));
        }

        @Override
        public void processElement(LedgerEvent ev, Context ctx, Collector<ReconResult> out)
                throws Exception {

            long now = ctx.timestamp() != null ? ctx.timestamp() : System.currentTimeMillis();
            if (now < ctx.currentWatermark() - lateThresholdMs) {
                // Drop events that are too late; route to a side output in production.
                return;
            }

            Double balance = balanceState.value();
            if (balance == null) balance = 0.0;

            String counterpartType = "CREDIT".equals(ev.type) ? "DEBIT" : "CREDIT";
            ListState<LedgerEvent> counterpartList =
                    "CREDIT".equals(ev.type) ? pendingDebits : pendingCredits;

            boolean matched = false;
            for (LedgerEvent candidate : counterpartList.get()) {
                if (Double.compare(candidate.amount, ev.amount) == 0) {
                    ReconResult r = new ReconResult(
                            ev.accountId, ev.reference, ReconStatus.MATCH,
                            balance, ev.amount, now);
                    out.collect(r);
                    counterpartList.remove(candidate);
                    matched = true;
                    break;
                }
            }

            if (!matched) {
                // No exact match; record pending and flag as missing-counterpart.
                if ("DEBIT".equals(ev.type)) pendingDebits.add(ev);
                else                          pendingCredits.add(ev);

                double expected = balance + ("CREDIT".equals(ev.type) ? ev.amount : -ev.amount);
                ReconStatus s = "DEBIT".equals(ev.type)
                        ? ReconStatus.MISSING_CREDIT
                        : ReconStatus.MISSING_DEBIT;
                out.collect(new ReconResult(
                        ev.accountId, ev.reference, s, expected, ev.amount, now));
            }

            // Update running balance (idempotent: balance = prior + credit - debit).
            double newBalance = balance + ("CREDIT".equals(ev.type) ? ev.amount : -ev.amount);
            balanceState.update(newBalance);
        }
    }

    // ---- Entry point ----
    public static void main(String[] args) throws Exception {

        StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();
        env.enableCheckpointing(60_000, CheckpointingMode.EXACTLY_ONCE);
        env.getCheckpointConfig().setMinPauseBetweenCheckpoints(30_000);
        env.getCheckpointConfig().setExternalizedCheckpointRetention(
                CheckpointConfig.ExternalizedCheckpointRetention.RETAIN_ON_CANCELLATION);
        env.getConfig().setParallelism(4);

        KafkaSource<String> kafka = KafkaSource.<String>builder()
                .setBootstrapServers("kafka:9092")
                .setGroupId("ledger-recon")
                .setTopics("ledger.events")
                .setStartingOffsets(OffsetsInitializer.earliest())
                .setValueOnlyDeserializer(new SimpleStringSchema())
                .build();

        DataStream<LedgerEvent> events = env
                .fromSource(kafka, WatermarkStrategy
                                .<String>forBoundedOutOfOrderness(Duration.ofSeconds(30))
                                .withTimestampAssigner((s, ts) -> {
                                    try {
                                        return new ObjectMapper()
                                                .readValue(s, LedgerEvent.class).eventTime;
                                    } catch (Exception e) { return 0L; }
                                }),
                        "kafka-ledger-source")
                .map(s -> new ObjectMapper().readValue(s, LedgerEvent.class))
                .returns(Types.POJO(LedgerEvent.class));

        DataStream<ReconResult> results = events
                .keyBy(e -> e.accountId)
                .process(new ReconciliationFunction(Duration.ofSeconds(30).toMillis()))
                .name("reconcile-per-account");

        // Two-phase-commit sink -> PostgreSQL
        results.addSink(JdbcSink.exactlyOnceSink(
                "INSERT INTO reconciliation_results " +
                        "(account_id, reference, status, expected_balance, observed_amount, processed_at) " +
                        "VALUES (?, ?, ?, ?, ?, ?) " +
                        "ON CONFLICT (account_id, reference) DO NOTHING",
                (ps, r) -> {
                    ps.setString(1, r.accountId);
                    ps.setString(2, r.reference);
                    ps.setString(3, r.status.name());
                    ps.setDouble(4, r.expectedBalance);
                    ps.setDouble(5, r.observedAmount);
                    ps.setLong(6, r.processedAt);
                },
                JdbcExactlyOnceOptions.builder()
                        .withTransactionPerConnection(true)
                        .build(),
                () -> {
                    // In real code: build a HikariCP pool + PGXADataSource
                    // and pass it via JdbcConnectionProvider with XA support.
                    throw new UnsupportedOperationException(
                            "Configure PGXADataSource-backed XAConnectionFactory here");
                }
        )).name("jdbc-sink");

        env.execute("ledger-reconciliation");
    }
}

## Dry Run

Input events for `accountId = A1`, in order:

```
E1: { type: CREDIT, amount: 100, reference: T1, eventTime: 1000 }
E2: { type: DEBIT,  amount: 100, reference: T1, eventTime: 1100 }
E3: { type: CREDIT, amount: 50,  reference: T2, eventTime: 1200 }
E4: { type: DEBIT,  amount: 30,  reference: T3, eventTime: 1300 }  // no matching credit
```

Step-by-step:

- E1 (CREDIT 100): pending debits empty, no match. Added to `pendingCredits`. Emits `MISSING_CREDIT`, `expectedBalance=100`, `balanceState=100`.
- E2 (DEBIT 100): scans `pendingCredits`, finds T1 with amount 100. Match. Emits `MATCH`, removes T1 from `pendingCredits`. `balanceState=100-100=0`.
- E3 (CREDIT 50): no match, added to `pendingCredits`. Emits `MISSING_CREDIT`, `expectedBalance=50`, `balanceState=50`.
- E4 (DEBIT 30): scans `pendingCredits`, no 30. Emits `MISSING_CREDIT`, `expectedBalance=50-30=20`, `balanceState=20`. Held in `pendingCredits` for T3? No, T3 is a debit, so it goes to `pendingDebits` to wait for a matching CREDIT 30.

## Complexity

- Time Complexity: O(N · K) per checkpoint window, where N is events per key and K is the average size of the counterpart list scanned per event. With indexed state (e.g., a `MapState<amount, List<reference>>`) the scan becomes O(N · log K) or O(N) amortized.
- Space Complexity: O(U) per key, where U is the number of unmatched pending events for that key (bounded in practice; needs a sweeper/timer for true unboundedness).

## Interviewer Follow-up Points

- Late events: instead of dropping, route to a side output for DLQ investigation, and add an `allowedLateness` / state TTL strategy.
- State growth: add a `ValueState<Long> lastUpdateTs` and register an event-time timer to clean up stale pending lists after a configurable window.
- Exactly-once: requires the sink to participate in Flink's two-phase commit; a `TwoPhaseCommitSinkFunction` or `JdbcSink.exactlyOnceSink` with an XA `DataSource` is mandatory. Non-XA JDBC is at-least-once and needs an idempotency key in the sink table.
- Determinism: balance updates are functions of the ledger, so a replay from Kafka after restoring from a checkpoint reproduces the same results, which is what you want for auditability.
- Hot keys: very active accounts can be skewed; consider splitting or local aggregation before the keyed process function.
- Serialization: using `Types.POJO` requires a no-arg constructor and public fields (or getters/setters); switch to `Avro` or `Kryo` for production performance and schema evolution.
- Watermark alignment: for multi-source joins, align watermarks across Kafka topics so a slow source doesn't block reconciliation of the faster one.
